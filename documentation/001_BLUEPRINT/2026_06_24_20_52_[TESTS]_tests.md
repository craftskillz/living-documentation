Le dossier `tests/` contient la suite de vérification du projet. C'est l'endroit où sont décrits les comportements attendus de Living Documentation, côté API, interface navigateur et logique isolée.

Il sert à vérifier que l'application continue de fonctionner quand le code évolue : création et lecture de documents, configuration, MCP, métadonnées, diagrammes, fichiers, images, édition, navigation et parcours utilisateur importants.

L'objectif est simple : sécuriser les fonctionnalités visibles du produit avec des scénarios reproductibles, lancés sur des environnements de test isolés.

Vue d'ensemble : `api/` vérifie les routes et le MCP, `e2e/` vérifie les parcours navigateur, `unit/` couvre quelques comportements ciblés, `fixtures/` fournit des projets de test, et `helpers/` prépare les instances isolées utilisées par les specs.

Fonctionnellement, ce dossier couvre cinq usages importants :

- valider les API et le serveur MCP ;
- vérifier les parcours utilisateur dans le navigateur ;
- tester des comportements ciblés sans lancer toute l'interface ;
- fournir des jeux de documents réalistes pour les scénarios ;
- démarrer et nettoyer des instances de test isolées pour éviter les effets de bord.
