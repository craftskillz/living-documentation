# Style de rédaction des documents Blueprint

Les documents Blueprint servent à cartographier humainement les dossiers importants du projet.

Principes à suivre :

- rédiger au niveau fonctionnel, pas au niveau implémentation ;
- expliquer à quoi sert le dossier pour un lecteur humain ;
- éviter les détails de tests, packaging, contrats internes et options techniques sauf s'ils expliquent une fonctionnalité visible ;
- ne pas écrire de section négative du type "ce que ce dossier ne contient pas" ;
- ne pas écrire "son rôle n'est pas", mais formuler directement "son rôle est" ou "il sert à" ;
- éviter les sections "points de vigilance" quand elles relèvent surtout de maintenance technique ;
- garder les sections courtes, lisibles et orientées usage ;
- mentionner les fichiers seulement pour identifier leur responsabilité fonctionnelle.
